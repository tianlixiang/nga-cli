# 05. Scale 阶段 — 从单兵到 agentic 团队

> **退出标准**：关键业务流程在创始人不在场时依然能跑通，并且团队（人 + agent）每周都能交付。

规模化阶段不再是"创始人 + AI"，而是"创始人 + 团队 + AI"。这里出现了 AI 原生公司最反直觉的事实：**招的第一个人不一定是工程师**。因为 AI 已经把工程产能拉满了，瓶颈在别的地方——销售、客户成功、运营。

## 阶段目标

1. **从"创始人英雄主义"切到可复制的流程**
2. **决定哪些事招人做、哪些事招 agent 做**
3. **建立可持续的财务模型**（不再靠融资续命）
4. **保住 PMF**，因为规模化最容易把 PMF 搞丢

## 招人 vs 招 agent 决策框架

每个新增工作，先问三个问题：

1. **它是判断密集，还是执行密集？**
   - 执行密集 → agent
   - 判断密集 → 人
2. **它需要客户信任吗？**
   - 需要 → 人（关键销售、客户成功）
   - 不需要 → agent
3. **错了的代价多大？**
   - 高代价（合同、产品方向） → 人
   - 低代价（写初稿、整理数据） → agent

### 典型分工

| 工作类型 | 谁做 | 原因 |
|---------|------|------|
| 写销售邮件初稿 | Agent | 执行密集，错了改就行 |
| 谈年合同 | 人（创始人 / 销售头） | 高代价 + 需要信任 |
| 写技术文档 | Agent + 工程师审 | Agent 出稿，人保证准 |
| 客户 onboarding | 人（C 端可以 agent） | B2B 需要信任，C 端不需要 |
| 数据看板 | Agent + 工程师审架构 | 执行密集 |
| 招聘面试 | 人 | 判断密集 + 客户信任的反向 |

## 财务模型

Scale 阶段必须切到"营收驱动"，不再靠融资续命：

### 三个数你必须每周看

1. **MRR（月度经常收入）**：增长率，最好月环比 ≥ 15%
2. **CAC payback 时间**：获客成本多久能回本，应 ≤ 12 个月
3. **现金跑道**：手上的钱够烧多久，应 ≥ 12 个月

### AI 原生公司的财务优势

| 项 | 传统 SaaS | AI 原生 |
|----|----------|--------|
| 工程师人头 | 高 | 低（AI 替代） |
| 客户成功人头 | 高 | 中（部分 agent 化） |
| 营销内容产出 | 慢 + 贵 | 快 + 便宜 |
| 毛利率 | 70-80% | 80-90% |

**毛利率 +10% 是巨大的优势**——意味着同样营收下你能投入更多到产品和销售。

## 保住 PMF

Scale 阶段最大的隐性风险：**你以为在 Scale 已有的 PMF，其实在悄悄丢**。

### 三个早期信号

1. **新客户留存下滑**（D30 从 40% 跌到 30%）
2. **客服 ticket 类型变化**（从 "怎么用" 变成 "为什么这样设计"）
3. **创始人不再亲自见客户**（最危险）

### 怎么守

- **创始人每周至少见 5 个客户**，无论公司多大
- **客户反馈每周一次走读**，agent 分类，创始人亲自看 top 10
- **不为 Scale 而 Scale**——增长速度低于团队膨胀速度时，立即收缩

## 常见失败模式

| 失败模式 | 原因 | 怎么破 |
|---------|------|--------|
| **过早招高级工程师** | "我们要 scale 工程" | AI 已经替了 80%，先招销售/CS |
| **创始人退到二线** | "我要做战略" | 还没到时候，继续亲自见客户 |
| **靠融资续命** | "再融一轮就好" | Scale 阶段必须营收驱动 |
| **复制硅谷打法** | "OpenAI/Anthropic 那么干" | 你不是 OpenAI，你是你 |
| **agent 跑无人区** | 全自动化，没人审 | 关键决策点必须有人 |

## 退出 Scale 阶段（或承认还没到）的检查清单

- [ ] MRR 月环比 ≥ 15%（连续 6 个月）
- [ ] CAC payback ≤ 12 个月
- [ ] 现金跑道 ≥ 12 个月
- [ ] 关键流程文档化，新人 1 周能上手
- [ ] 创始人每周仍亲自见 ≥ 5 个客户
- [ ] D30 留存没有下滑

如果有 2 项以上未达标，**回到 Launch 阶段**重新打磨。Scale 是结果不是目标。
